# 需求分析文档

> 项目：Mihomo 多网段精细分流配置  
> 版本：v6  
> 最后更新：2026-04-12

---

## 1. 项目背景

用户拥有一台运行 OpenWrt 的路由器，已安装 Mihomo（原 Clash Meta）插件。需要实现一套**多网段、多机场、精细分流**的代理配置，满足全家多设备安全上网需求。

## 2. 用户画像

- 技术水平：熟悉路由器基础操作，了解代理概念，但非网络工程师
- 设备情况：多台手机/电脑/平板，通过不同 WiFi AP 接入
- 使用场景：AI 开发、海外社交、流媒体观看、设计工作、加密货币交易、游戏
- 核心诉求：**一次配置，全家受益，不同 WiFi 不同策略**

## 3. 功能需求

### 3.1 多网段隔离

| 需求编号 | 网段 | WiFi AP 名称 | 行为 | 优先级 |
|---------|------|-------------|------|--------|
| NET-01 | 192.168.31.x | ap-direct / br-lan | 全部直连，不经过任何代理 | P0 |
| NET-02 | 192.168.32.x | ap-rule | 按规则精细分流 | P0 |
| NET-03 | 192.168.33.x | ap-Global | 全部流量强制走代理 | P0 |
| NET-04 | 192.168.34.x | ap-34x | **默认**走静态住宅 IP；**白名单**（如远程桌面相关域名/端口/IP）走直连 | P0 |
| NET-05 | 192.168.35.x ~ 255.x | 待扩展 | 链式代理/其他住宅 IP；命名与规则模式对齐 v6 住宅方案 | P2 |

### 3.1.1 住宅网段白名单（v6）

| 需求编号 | 描述 | 优先级 |
|---------|------|--------|
| RES-01 | 源为住宅网段（如 34.x）时：命中白名单的目的流量 **DIRECT**，其余仍走该网段住宅出口 | P0 |
| RES-02 | 动机：远程连接经住宅代理会导致延迟过高；仅对白名单「借道」国内直连出口 | P0 |
| RES-03 | 白名单：`DIRECT-34-relays.yaml`（端口/IP）+ `DIRECT-34.yaml`（域名），由 `v6.yaml` 中 `SUB-RULE` + 两条 `RULE-SET` 引用 | P0 |

| 需求编号 | 分类 | 包含平台 | 优先级 |
|---------|------|---------|--------|
| APP-01 | AI 平台 | Claude, ChatGPT, Gemini, DeepSeek | P0 |
| APP-02 | 视频流媒体 | YouTube, Netflix, TikTok, Spotify, Disney+ | P0 |

### 3.3 多机场管理

| 需求编号 | 描述 | 优先级 |
|---------|------|--------|
| AIR-01 | 支持 5 个机场订阅，按梯队分组，自动拉取更新 | P0 |
| AIR-02 | 三梯队分层：C 主力（2机场）> B 保底（2机场）> A 优质（1机场） | P0 |
| AIR-03 | 视频/社交类走 C→B→A（主力先上，省流量优先） | P0 |
| AIR-04 | AI/重要业务走 A→C→B（优质先上，稳定优先） | P0 |
| AIR-05 | 某机场掉线时自动故障转移到下一个 | P0 |
| AIR-06 | 按地区筛选节点（港/日/新/美/台/英/德/法/韩） | P0 |
| AIR-07 | 自动过滤机场展示/说明/流量提示等无用节点 | P1 |

### 3.4 策略组 UX 需求

| 需求编号 | 描述 | 优先级 |
|---------|------|--------|
| UX-01 | 策略组名称必须标注所属网段（如 `Claude 32.x`） | P0 |
| UX-02 | 应用策略组只显示国家/地区名，不显示机场/故转/全部节点 | P0 |
| UX-03 | 面板排列顺序：出口组 → 应用组 → 地区组 → 机场子组 | P0 |
| UX-04 | AI 大类拆分为独立平台（不要笼统的"AI 平台"） | P0 |
| UX-05 | 所有故转合并为一个统一的故障转移组 | P1 |

### 3.5 DNS 安全

| 需求编号 | 描述 | 优先级 |
|---------|------|--------|
| DNS-01 | 使用 fake-ip 模式防止 DNS 泄漏 | P0 |
| DNS-02 | 国内域名走国内 DOH（阿里/腾讯），海外域名走 8.8.8.8/1.1.1.1 | P0 |
| DNS-03 | 代理节点域名解析不经过代理（防鸡蛋问题） | P0 |

## 4. 非功能需求

| 需求编号 | 描述 | 优先级 |
|---------|------|--------|
| NFR-01 | 配置文件可扩展：新增平台/网段只需添加几行 | P1 |
| NFR-02 | 规则集优先使用 .mrs 二进制格式（加载快、体积小） | P1 |
| NFR-03 | 敏感信息（订阅 URL/住宅 IP 凭据）集中在文件顶部，便于管理 | P1 |
| NFR-04 | 控制面板 (zashboard) 可远程管理策略 | P2 |
| NFR-05 | 本仓 rulesets：**文件名** = `策略简写-网段`（住宅白名单可拆 `DIRECT-34-relays.yaml` + `DIRECT-34.yaml`）；**provider 键** = 小写+下划线（如 `direct_34_relays`、`direct_34`） | P0 |

## 5. 约束条件

- 运行平台：OpenWrt + Mihomo 插件
- 内核版本：Mihomo (Clash Meta) 最新稳定版
- 硬件限制：路由器内存有限，规则集不宜过多
- 网络环境：中国大陆，部分域名/IP 被墙

## 6. 验收标准

1. 连接 ap-direct 的设备 100% 直连，无代理流量
2. 连接 ap-rule 的设备按规则分流，各平台走对应策略组
3. 连接 ap-Global 的设备 100% 走代理
4. 连接 ap-34x：未命中 `direct_34_relays` / `direct_34` 的流量走住宅 IP；命中任一则直连
5. 视频平台优先使用最便宜机场，AI 平台优先使用最好机场
6. 单机场掉线后 300 秒内自动切换到下一个机场
7. 面板中应用策略组排列在地区组之前
8. 所有规则集 URL 返回 HTTP 200
